### #2 commit-message is a shell-injection vector (action.yml:129)

**Status: fixed**

The commit step runs:

```bash
git commit -q -m "$COMMIT_MESSAGE"
```

`commit-message` is a user-controlled input. Inside double quotes, `$(...)`
and backticks still execute, so a malicious or accidental value could trigger
shell expansion or break the commit.

**Fix:** pass the message via stdin or a file so the shell never interprets it:

```bash
printf '%s\n' "$COMMIT_MESSAGE" | git commit -q -F -
```

Add a test (or end-to-end check) that a message containing `$(...)` or
backticks is committed verbatim.

### #3 letter-spacing not accounted for in badge width (renderer.js:122)

**Status: fixed**

The `for-the-badge` style adds `letter-spacing="1"` to the `<text>` element
(via `styleGeometry`, renderer.js:79) but `textWidth` (renderer.js:68-70)
ignores it. Both the label and the value get spaced, so the computed width is
short by roughly `(labelLen - 1) + (valueLen - 1)` pixels. With longer labels
at larger font sizes the text overflows the colored value rect.

**Fix:** thread `letterSpacing` through to the width formula:

```js
function textWidth(text, fontSize, letterSpacing = 0) {
  const charWidth = text.length * fontSize * CHAR_WIDTH_RATIO;
  const spacing = Math.max(0, text.length - 1) * letterSpacing;
  return charWidth + spacing;
}
```

Pass `geom.letterSpacing || 0` from `render`. TDD: add a renderer test that
the `for-the-badge` style produces a wider badge than `flat` for the same
label and font-size, beyond what font-size alone explains.

## Not a bug

### #1 Token in git remote URL (action.yml:86)

**Status: not a bug**

The original review flagged this as critical, claiming the PAT could leak
through subprocess error output. GitHub Actions masks secrets in *all* log
output (including subprocess stderr) as long as they flow through
`${{ inputs.token }}`, which they do here via the `TOKEN` env var.

Switching to the `extraheader` pattern (as `actions/checkout` does) would be
a stylistic cleanup — it keeps the token out of `git remote -v` — but it is
not closing a leak. Not scheduled.

## Minor (not scheduled)

### #4 SVG missing `<title>` for accessibility (renderer.js:132)

The SVG has `role="img"` and `aria-label` but no `<title>`. Some screen
readers prefer `<title>` in SVG context. Adding
`<title>${labelSafe}: ${valueSafe}</title>` as the first child is two lines.

### #5 Width heuristic underestimates wide characters (renderer.js:68-70)

`CHAR_WIDTH_RATIO = 0.6` is a documented approximation. A per-character
Verdana lookup table (like shields.io) would be more accurate but is a lot
of code for a known heuristic the project explicitly accepts.

### #6 No output path validation (options.js:167-169)

`output` is passed through verbatim. A value like `../../etc/garbage` could
write outside the expected directory inside `RUNNER_TEMP`. The Action runs
in the user's own repo on their own input, so the risk is low.

### #7 No SVG structural validation in tests (test/renderer.test.js)

Tests check substrings and basic `<svg>` presence but don't verify rect/text
geometry or XML well-formedness. Note: Node 20 does **not** expose
`DOMParser` on `globalThis`, so the original suggestion to use it without a
dependency does not work — adding a parser would violate the zero-dep rule.

### #8 normalizeState silently tolerates corrupted state (accumulator.js:19-26)

If `totals.json` is hand-edited so `views`/`clones` are arrays instead of
objects, `normalizeState` recovers silently. Logging a warning on unexpected
shape would surface the corruption without changing behaviour.
